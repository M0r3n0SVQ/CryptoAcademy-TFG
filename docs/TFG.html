1. Casos de Uso

En esta sección se detallan los casos de uso que definen las interacciones fundamentales entre los actores y el sistema CryptoAcademy. Cada caso de uso representa una funcionalidad clave que el sistema debe proporcionar.

1.1. Gestión de Cuentas

CU-01: Registrar Nuevo Usuario

ActorVisitante (Usuario no autenticado)
DescripciónPermite a un nuevo visitante crear una cuenta en el sistema para acceder a sus funcionalidades.
PrecondicionesEl visitante no debe tener una cuenta existente con el mismo correo electrónico.
Flujo Principal 1. El visitante accede a la pantalla de registro.
2. Rellena el formulario con su nombre, correo electrónico y contraseña.
3. El sistema valida que los datos son correctos y el email no existe.
4. El sistema crea una nueva cuenta de usuario y una cartera inicial asociada con 10,000 USD simulados.
5. El sistema autentica al nuevo usuario y le redirige a la pantalla principal.
Flujos Alternativos 3a. Error de validación: Si el email no es válido o la contraseña es insegura, el sistema muestra un mensaje de error.
4a. Email duplicado: Si el email ya está registrado, el sistema informa al usuario.
PostcondicionesSe ha creado una nueva cuenta de usuario en la base de datos y el usuario ha iniciado sesión.

CU-02: Iniciar Sesión

ActorUsuario registrado
DescripciónPermite a un usuario registrado acceder a su cuenta.
PrecondicionesEl usuario debe tener una cuenta registrada y activa en el sistema.
Flujo Principal 1. El usuario accede a la pantalla de inicio de sesión.
2. Introduce su correo electrónico y contraseña.
3. El sistema verifica las credenciales.
4. El sistema autentica al usuario y le redirige a la pantalla principal.
Flujos Alternativos 3a. Credenciales incorrectas: Si el email o la contraseña no coinciden, el sistema muestra un mensaje de error.
PostcondicionesEl usuario ha iniciado sesión y tiene acceso a las funciones personalizadas.

1.2. Trading Simulado

CU-03: Comprar Criptomoneda

ActorUsuario autenticado
DescripciónPermite a un usuario comprar una cantidad de una criptomoneda utilizando su saldo simulado en USD.
PrecondicionesEl usuario debe haber iniciado sesión y tener una cartera activa con saldo suficiente en USD.
Flujo Principal 1. El usuario selecciona una criptomoneda del mercado.
2. El usuario elige la opción "Comprar" y especifica la cantidad a adquirir.
3. El sistema calcula el coste total de la operación basado en el precio de mercado actual.
4. El sistema verifica que el usuario tiene saldo suficiente en su cartera.
5. El sistema deduce el coste de la cartera, añade la criptomoneda a sus tenencias y registra la transacción.
6. El sistema notifica al usuario que la compra se ha realizado con éxito.
Flujos Alternativos 4a. Saldo insuficiente: Si el usuario no tiene fondos suficientes, el sistema cancela la operación y muestra un mensaje de error.
PostcondicionesEl saldo en USD del usuario ha disminuido y su tenencia de la criptomoneda comprada ha aumentado.

CU-04: Vender Criptomoneda

ActorUsuario autenticado
DescripciónPermite a un usuario vender una cantidad de una criptomoneda que posee.
PrecondicionesEl usuario debe haber iniciado sesión y poseer la cantidad de la criptomoneda que desea vender.
Flujo Principal 1. El usuario selecciona una criptomoneda de su portafolio.
2. El usuario elige la opción "Vender" y especifica la cantidad.
3. El sistema verifica que el usuario posee la cantidad especificada.
4. El sistema deduce la criptomoneda de las tenencias del usuario, incrementa su saldo en USD y registra la transacción.
5. El sistema notifica al usuario que la venta se ha completado.
Flujos Alternativos 3a. Tenencia insuficiente: Si el usuario intenta vender más de lo que posee, el sistema cancela la operación y muestra un error.
PostcondicionesLa tenencia de la criptomoneda del usuario ha disminuido y su saldo en USD ha aumentado.

1.3. Consulta de Información

CU-05: Consultar Portafolio

ActorUsuario autenticado
DescripciónPermite al usuario visualizar el estado actual de su cartera, incluyendo el valor total y el desglose de sus activos.
PrecondicionesEl usuario debe haber iniciado sesión.
Flujo Principal 1. El usuario navega a la sección "Portafolio".
2. El sistema recupera todas las tenencias de la cartera activa del usuario.
3. El sistema obtiene los precios de mercado actuales de cada criptomoneda.
4. El sistema calcula el valor actual de cada tenencia y el valor total del portafolio.
5. El sistema muestra la información de forma clara, incluyendo gráficos y porcentajes de distribución.
PostcondicionesEl usuario conoce el valor y la composición de sus inversiones simuladas.

2. Diagrama de Clases

2.1. Clases Principales del Backend

La estructura del backend se organiza en las siguientes capas y clases principales:

Entidades JPA (Persistencia):

  • Usuario: Representa al usuario y sus datos.
  • Cartera: Almacena el saldo y las tenencias de un usuario.
  • Criptomoneda: Guarda los datos de mercado de una cripto.
  • Transaccion: Registra una operación de compra/venta.
  • CriptosAlmacenadas: Tabla de unión para las tenencias.
  • Enums: RolUsuario, TipoTransaccion.

DTOs y Servicios:

  • DTOs: Clases como PortfolioResponseDTO o RankingItemDTO para la comunicación.
  • Servicios: Clases como TradingServiceImpl o AuthService que contienen la lógica de negocio.

2.2. Diagrama Visual

Diagrama UML que muestra la estructura estática y las relaciones entre las clases principales del backend.

3. Diagrama Entidad-Relación (DER)

3.1. Entidades Principales y Relaciones

El DER se basa directamente en las entidades JPA y cómo se mapean a tablas de base de datos. Las relaciones clave aseguran la integridad referencial:

  • Un usuario tiene múltiples carteras y transacciones.
  • Una cartera pertenece a un único usuario.
  • Una transaccion está asociada a un usuario, una cartera y una criptomoneda.
  • criptos_almacenadas resuelve la relación N:M entre carteras y criptomonedas.
  • audit_log_transacciones mantiene un registro histórico de las transacciones.

3.2. Diagrama Visual

Diagrama generado desde MySQL Workbench que muestra el esquema físico de la base de datos.

4. Diagrama de Componentes

4.1. Componentes Principales del Sistema

La arquitectura de CryptoAcademy se divide en los siguientes componentes principales:

  • Aplicación Móvil (Flutter): Componente cliente responsable de la UI/UX y la interacción con el usuario.
  • API Backend (Spring Boot): Componente servidor que contiene la lógica de negocio, seguridad y acceso a datos.
  • Base de Datos (MySQL): Componente de persistencia para todos los datos de la aplicación.
  • API Externa (CoinGecko): Servicio externo que provee los datos de mercado en tiempo real.

5. Casos de Prueba

A continuación se resumen los casos de prueba ejecutados para verificar las funcionalidades clave del sistema.

5.1. Pruebas del Backend (API)

Caso de Prueba Descripción Resultado Esperado
Registro de usuario válido Registrar un usuario con datos correctos y únicos. Código 201, token JWT generado, usuario creado en BD.
Registro con email duplicado Intentar registrar con un email ya existente. Código 400, mensaje de error "Email ya en uso".
Login válido Iniciar sesión con credenciales correctas. Código 200, token JWT válido devuelto.
Login con credenciales incorrectas Iniciar sesión con contraseña errónea. Código 401, mensaje de error "Credenciales inválidas".
Compra válida Comprar criptomoneda con saldo suficiente. Código 200, transacción creada, saldos actualizados.
Compra con fondos insuficientes Intentar comprar con saldo insuficiente. Código 400, mensaje de error "Saldo insuficiente".
Venta válida Vender criptomoneda con tenencia suficiente. Código 200, transacción creada, saldos actualizados.
Venta con tenencia insuficiente Intentar vender más criptomonedas de las que se poseen. Código 400, mensaje de error "Tenencia insuficiente".
